Крутая идея: перенести A-Maze-ing в веб и сделать из него 2D игру с красивой графикой, управлением с клавиатуры и мобильным джойстиком.

Ниже — уже нормальное ТЗ для ИИ, чтобы он не просто “сделал что-нибудь”, а собрал понятную игру с правильной архитектурой.

## ТЗ для веб-игры Urduliz Maze Adventure

### 1. Цель проекта

Сделать 2D веб-игру про прохождение процедурно сгенерированного лабиринта в стиле сад-лабиринта. Игрок управляет персонажем, ищет выход, может получить подсказку кратчайшего пути и видеть seed текущего лабиринта.

### 2. MVP: что обязательно должно быть в первой версии

* Генерация лабиринта по seed.
* Размер карты настраивается в меню.
* Игровое поле отображается в 2D.
* Есть вход, выход и персонаж игрока.
* Есть центральная зона 7x5 с логотипом 42.
* Есть управление с клавиатуры.
* Есть управление для мобильных устройств.
* Есть экран меню и экран игры.
* Есть кнопка подсказки пути.
* Есть кнопка копирования seed.

### 3. Технический стек

* Язык: TypeScript.
* Рендер: HTML5 Canvas (чистый, без React, для лучшей производительности).
* UI: HTML и CSS для меню поверх Canvas.
* Стили: CSS.
* Логика генерации: отдельный модуль, не смешивать с рендерингом.
* Сборка: Webpack или простой TypeScript компилятор.

### 4. Основная игровая логика

* Использовать DFS Recursive Backtracker как первый алгоритм генерации.
* Позже архитектура должна позволять добавить другие алгоритмы.
* Размеры лабиринта: от 15x15 до 50x50 клеток.
* Seed должен полностью определять генерацию карты.
* Вход и выход не должны попадать в центральную зону 42.
* **Perfect-режим включен в первой версии**: лабиринт генерируется без циклов (идеальный лабиринт — дерево прохождения).
* Для кнопки Path использовать BFS и подсвечивать кратчайший путь от текущей позиции игрока до выхода.
* **Вход и выход расставляются автоматически на краях карты**: вход верхний-левый, выход нижний-правый. Это упрощает генерацию и валидацию.

### 5. Зона 42

* В центре карты должна быть зарезервирована зона 7x5 клеток.
* Внутри этой зоны отображается логотип 42.
* Эта зона не должна быть перекрыта стенами, входом, выходом или спавном игрока.
* Вокруг зоны должны быть непроходимые границы, чтобы она выглядела как отдельный защищенный объект внутри лабиринта.

### 6. Визуальный стиль

* Общий стиль: красивый сад-лабиринт, вдохновленный референсом.
* Ссылка на референс: https://urdulizlinkup.org/wp-content/uploads/lan-map/
* Пол: каменные дорожки, утоптанная земля или другой читаемый светлый тайл.
* Стены: плотные кусты, цветочные поля или декоративная зелень, чтобы они контрастировали с полом.
* Внешняя область за пределами лабиринта: трава или природный фон.
* Игрок: анимированный персонаж девочки.
* Выход: магический портал, сияние или другой заметный визуальный эффект.
* Декор: в тупиках и закрытых участках можно добавить редких анимированных животных или птиц.

### 7. Управление

* Desktop: стрелки и/или WASD.
* Mobile: экранный джойстик или ползунок движения.
* Кнопка Path: включает подсветку маршрута до выхода.
* Кнопка Back to Menu: возвращает в главное меню.

### 8. Интерфейс

#### Главное меню (HTML + CSS поверх Canvas)

* Поле Seed: если пустое, генерируется случайный seed.
* Кнопка Paste/Apply для вставки seed из буфера, если он уже есть.
* Размер Width: минимум 15, максимум 50 (spinner или слайдер).
* Размер Height: минимум 15, максимум 50 (spinner или слайдер).
* **Perfect-режим**: всегда включен (не выключается в MVP).
* Выбор алгоритма оставить как заготовку на будущее (сейчас только DFS).
* Кнопка Start / Play.

#### HUD в игре

* Seed отображается в левом верхнем углу.
* Рядом с seed есть кнопка Copy.
* Есть кнопка "Path Hint" (показать кратчайший путь).
* Есть кнопка Back to Menu.

#### На вторую версию

* **Звук и музыка**: отложены на вторую версию, не критичны для MVP.
* Выбор входа и выхода в меню.
* Toggle для режима Perfect.

### 9. Ограничения и правила

* Нельзя использовать слишком тяжелую DOM-сетку из тысяч div для поля 50x50, лучше Canvas.
* Все координаты и проверки должны быть валидны относительно размеров карты.
* Выход не должен совпадать с входом.
* Автоматические координаты входа и выхода лучше делать на краях карты, если ручной выбор окажется слишком сложным для первой версии.

### 10. Хорошая архитектура для кода

* MazeGenerator — генерация карты.
* MazeState — состояние игры, seed, размеры, позиция игрока, вход, выход.
* PathFinder — поиск пути.
* Renderer — отрисовка карты и объектов.
* InputController — клавиатура и мобильное управление.
* UI Layer — меню, кнопки, настройки.

### 11. Что можно отложить на вторую версию

* Звуки и музыка.
* Дополнительные алгоритмы генерации (Prim, Eller и т.д.).
* Сложная анимация персонажей.
* Toggle для режима Perfect (он всегда включен в v1).
* Ручной выбор входа и выхода в меню.
* Дополнительные украшения карты (животные в тупиках).

### 12. Финальные решения

✅ **Стек**: Canvas + TypeScript (без React), HTML+CSS для меню.  
✅ **Perfect-режим**: Включен в первой версии (лабиринт без циклов).  
✅ **Вход и выход**: Автоматически на краях карты (вход верх-левый, выход низ-правый).  
✅ **Звук и музыка**: Отложены на вторую версию.
